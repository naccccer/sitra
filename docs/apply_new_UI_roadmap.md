- Rollout order is locked as `standards first -> module rollout waves`.

## Locked Visual Standards
### Table Family
- Match the orders workspace table density for admin list surfaces unless a documented exception is approved.
- Match the current orders header style: compact muted header text, strong separation, and consistent center/edge alignment.
- Match the current orders row hover behavior, selected-row behavior, and muted archived-row treatment.
- Match the current action-column treatment: centered action cluster, compact spacing, and orders-style white-surface controls.

### Action Language
- Prefer `IconButton` plus shared action presets for row-level actions.
- Use `Button` when the orders workspace uses a full button treatment for clarity or emphasis.
- Keep actions compact, RTL-safe, and visually consistent with the current orders workspace.
- Do not invent module-specific action iconography or ad hoc button stacks when a shared preset exists.

### Badge And Status Pill Treatment
- Reuse shared `Badge` semantics and the current orders status-pill language.
- Payment, workflow, and admin-state pills should read as members of the same family across modules.
- Do not introduce bespoke badge classes or one-off tone systems without documenting an approved exception.

### Persian Numerals And Formatting
- Use Persian numerals through `toPN` for visible counts, amounts, codes, identifiers, and other end-user numeric output where the orders reference does so.
- Standardize number presentation across list/detail/admin surfaces before new module waves continue.
- Preserve explicit `dir="ltr"` only for values that need LTR scanning, such as phone numbers, tracking/order codes, and similar tokens.

### Typography And Weight Hierarchy
- Preserve the current Vazirmatn-based system already active in the app.
- Match the orders workspace hierarchy for headers, body cells, meta text, and emphasized values.
- Use weight changes intentionally for scan hierarchy instead of introducing new ad hoc text scales.
- Prefer tabular-number treatment where it improves data scanning and fits the existing shared typography system.

### RTL Alignment Rules
- Keep RTL-first layout at the workspace boundary.
- Use RTL-safe alignment and spacing rules throughout list, detail, toolbar, and modal surfaces.
- Keep `dir` usage explicit only where the orders reference needs mixed-direction content.
- Do not introduce left/right-only assumptions that break logical alignment in Persian admin surfaces.

### Workspace Behavior
- Match the current orders conventions for:
  - toolbar composition
  - filter row composition
  - pagination placement
  - empty state
  - loading state
  - error state
  - detail expansion rhythm
- Preserve print-safety for sales/order-adjacent surfaces.

## Rollout Guardrails
- One Codex session per phase.
- Stop after the current phase is complete. Do not continue into the next phase in the same session.
- This rollout must stay within frontend/shared-doc scope unless doc drift requires a documentation correction.
- Do not add backend/API/schema work to any phase under this roadmap.
- Do not add cross-module private UI imports.
- Do not copy private sales internals into another module. Shared behavior belongs in `src/components/shared/ui/*` or module-local adapters.
- Shared primitive changes must stay neutral and reusable, not sales-specific in naming or assumptions.
- Module rollouts must use module-owned files under `src/modules/<module>/*`.
- Approved exceptions must be written into this roadmap before a phase is considered complete.
